# Updating the jobs board

The website builds `/jobs` from `job_postings.csv` at build time. It does not connect to the production database — this CSV is a committed export of the currently-active postings.

## How the CSV is produced

The CSV is generated by the sibling [`narrative-job-board`](../narrative-job-board) service, which scrapes the sources, upserts them into Neon Postgres, runs heartbeat (active/inactive) checks, de-duplicates across sources, and exports the active rows to this file. Two commands produce it:

- **`poetry run narrative-jobs`** — runs the pipeline and writes the CSV to the destination below, but does **not** commit or push. Keep this `game-writing.com` checkout on the `narrative-job-board` branch while you run it, then review, commit, and push the changed CSV yourself.
- **`poetry run publish-jobs`** — runs the pipeline and then commits and pushes the CSV for you onto the `narrative-job-board` branch of this repo. The worktree must be clean; the command switches to that branch, commits only the CSV, pushes, and returns to your original branch. Use this to skip the manual commit/push, then open or merge the branch.

The default destination is:

```text
../game-writing.com/src/data/jobs/job_postings.csv
```

Set `JOBS_CSV_PATH` in the service's `.env` to override it.

## What the build expects

The build publishes rows whose `is_active` value is `true`, `t`, `1`, or `yes`, and ignores the rest. The `description` column must contain valid JSON; the service's CSV export quotes and escapes it correctly. Every row must have the same number of columns as the header, or the build fails.

Required columns:

- `posting_id`
- `source`
- `company_name`
- `canonical_url`
- `title`
- `is_active`
- `first_seen_at`
- `last_updated`

The page also uses `project`, `location`, `employment_type`, `remote_type`, `writer_match_confidence`, and `description` when present. Any extra columns in the export (for example `job_board_id`) are ignored.

## Known deferral: country matching & character encoding (as of 2026-06-28)

**No code change has been made for this. It is a known, intentional deferral that needs a decision later.**

### The data is fine — don't "fix" the CSV

The exported `job_postings.csv` is clean UTF-8. Verified at the byte level: zero `U+FFFD` replacement characters, zero double-encoded ("mojibake") byte signatures, and accented place names are correctly encoded (e.g. `Montréal` → `… C3 A9 …`, `Malmö` → `… C3 B6 …`). Any garbled characters you might see in a terminal or diff (`MontrÃ©al`, `Malm�`) are your console/editor rendering UTF-8 through a non-UTF-8 codepage — **not** corruption in the file. Nothing about the CSV or how it's generated needs to change.

### The real issue is in `src/lib/jobs.ts`

`COUNTRY_MARKERS` hard-codes literal mojibake strings:

```ts
["Canada", ["canada", "montreal", "montrÃ©al", "quebec", "quÃ©bec"]],
["Sweden", ["sweden", "stockholm", "malmÃ¶", "malmo"]],
```

The `montrÃ©al` / `quÃ©bec` / `malmÃ¶` markers were added to match a **previous** export that emitted mojibake. The current export is clean UTF-8, so those markers now match **nothing** — they're dead code. Country facets still work today only because each location usually also contains an ASCII country token (`Canada`, `Poland`, …) that the plain markers catch.

### Why "just use the accented spelling" is NOT a reliable fix

The tempting fix is to replace `"montrÃ©al"` with `"montréal"`. That is brittle. `getJobCountries` matches with `String.includes()`, which compares Unicode **code points exactly**, and `é` has two valid encodings:

- **NFC** — one code point, `U+00E9` (what the CSV uses today)
- **NFD** — `e` + combining acute accent `U+0301`

They look identical but are different sequences, and `.includes()` will not match across them. So an accented NFC marker matches today's data but silently misses the moment an upstream source, a copy/paste, or a file re-save produces NFD — the exact brittleness that created the original mojibake markers.

### Recommended fix (when this is picked up)

Stop matching on accented characters. Strip diacritics on both sides and keep ASCII-only markers, so `Montréal` (NFC), `Montréal` (NFD), and `Montreal` all collapse to `montreal`:

```ts
// in getJobCountries, when building `normalized`:
const normalized = ` ${location
  .toLowerCase()
  .normalize("NFD")
  .replace(/[\u0300-\u036f]/g, "")   // strip combining accents
  .replace(/\s+/g, " ")} `;

// and make the markers ASCII-only:
["Canada", ["canada", "montreal", "quebec"]],
["Sweden", ["sweden", "stockholm", "malmo"]],
```

Caveat: a few letters don't decompose to ASCII (Polish `ł`, Danish `ø`), so `Wrocław` stays `wrocław`; those rows already match via their country token, so it's a non-issue for the current marker set.

### Decision needed

Pick one, later:

- **(a) Strip diacritics + ASCII markers** — recommended; future-proof, removes the whole class of bug.
- **(b) Swap mojibake markers for accented NFC forms** — cheap, but breaks on NFD input.
- **(c) Leave as-is** — country facets keep relying on the ASCII country token in each location; accent-only city matches won't fire.
